# 道路数据源指南

## 📌 两套道路系统架构

项目中存在**两套完全独立的道路系统**，请勿混淆：

| 系统 | 数据文件 | 渲染层变量 | 用途 |
|------|----------|------------|------|
| **手绘历史道路** | `src/data/VectorRoadData.ts` | `vectorRoadLayer` | 秦汉时期真实历史道路 |
| **Natural Earth 道路** | `public/assets/roads_filtered.geojson` | `globalRoadLayer` | 现代道路网络参考 |

### 系统1: 手绘历史道路 (主要)

- **文件**: `src/data/VectorRoadData.ts`
- **格式**: TypeScript GeoJSON FeatureCollection
- **内容**: 历史考证后手工绘制的道路（滏口陉、河内-邯郸道、太行八陉等）
- **特点**: 起点/终点必须使用 `cities.ts` 中的城市坐标

### 系统2: Natural Earth 道路 (参考)

- **文件**: `public/assets/roads_filtered.geojson`
- **来源**: Natural Earth 网站下载并筛选
- **内容**: 约17,000条现代亚洲道路
- **用途**: 仅作为绘制历史道路时的地理参考底图

> [!WARNING]
> Natural Earth 是**现代道路数据**，不可直接用于表示历史道路！
> 历史道路必须手动绘制到 `VectorRoadData.ts` 中。

## ⚠️ 道路设计规范（必读）

在绘制历史道路前，必须遵守以下规范：

### 1. 起点/终点必须使用城市坐标

道路的**起点和终点**必须是 `src/data/cities.ts` 中已定义城市的精确坐标。

```typescript
// ❌ 错误：使用随意坐标
coordinates: [[112.5, 36.0], ..., [114.3, 36.5]]

// ✅ 正确：使用 cities.ts 中的城市坐标
// city_changzi: [112.877562, 36.121345]
// city_handan: [114.4553, 36.5862]
coordinates: [[112.877562, 36.121345], ..., [114.4553, 36.5862]]
```

### 2. 途经据点必须先在 cities.ts 中定义

如果道路需要经过某个历史关隘或城池，该据点必须**先**在 `cities.ts` 中创建，然后才能在路线中引用其坐标。

### 3. 命名规范

| 禁止 | 现代行政区划名 | ❌ xx省、xx市 |

### 4. 道路类型

| type 值 | 用途 | 示例 |
|---------|------|------|
| `road` | 官道、主干道 | 秦直道、驰道 |
| `path` | 山间小径、陉道 | 太行八陉 |
| `plank` | 栈道 | 陈仓道、褒斜道 |

## 添加自定义历史道路

如果 Natural Earth 数据不包含某条历史道路 (如陈仓道)，可以手动添加:

1. 在 `src/data/VectorRoadData.ts` 中添加坐标
2. 设置 `type: 'plank'` (栈道) 或其他自定义类型
3. `VectorRoadLayer.ts` 会自动应用对应样式

**示例**:
```typescript
{
    type: "Feature",
    properties: {
        name: "陈仓道",
        type: "plank",
        startYear: -206,
        endYear: 200
    },
    geometry: {
        type: "LineString",
        coordinates: [[107.23, 34.37], [106.93, 33.07], ...]
    }
}
```

## 🛑 常见问题与解决方案 (Solutions)

### 1. 道路断开与直线问题 (Straight Line Issue)
- **现象**: 自动寻路生成的路径是直线，无视山脉和现有道路。
- **原因**: 起点和终点位于两个互不连通的路网孤岛（例如被太行山阻隔），Dijkstra 算法无法找到全路网路径。
- **解决方案**: **智能桥接 (Smart Bridging)**
    - 系统会自动探测起点可达区域 (Component A) 和终点可达区域 (Component B)。
    - 算法在两个区域的所有节点中，寻找**空间距离最近**的一对节点。
    - 生成混合路径：`路网A -> [直线桥接] -> 路网B`。
    - **效果**: 最大限度利用现有路网，仅在断路处生成最短的直线连接，模拟“翻山越岭”或“新建道路”。

### 2. 运行时错误 (Runtime Error)
- **现象**: `Cannot read properties of undefined (reading 'name')`
- **原因**: `VectorRoadData.ts` 中的数据可能包含空行、缺少属性或格式不完整（常见于手动复制粘贴时）。
- **解决方案**: **防御性编程**
    - 系统已在渲染层 (`SimpleVectorRoadRenderer`)、编辑器 (`VectorRoadEditor`) 和数据层 (`RoadRegistry`) 增加了全面的空值检查 `if (!feature || !feature.properties)`。
    - 任何格式错误的数据都会被自动跳过并打印警告，而不会导致游戏崩溃。

### 3. 可视化风格 (Visual Style)
- **方案**: **双层描边 (Cased Line)** 技术
- **配色**: 采用历史感强烈的大地色系 (Earth Tones)。
- **样式**:
    | 类型 | 内部颜色 | 外部描边 | 宽度 | 效果 |
    | :--- | :--- | :--- | :--- | :--- |
    | **官道 (road)** | #BCAAA4 (浅褐) | #5D4037 (深褐) | 3px / 5px | 宽阔主干道，清晰醒目 |
    | **次级 (secondary)** | #A1887F (中褐) | #4E342E (中褐) | 2px / 4px | 普通连接道路 |
    | **小径 (path)** | #8D6E63 (深褐) | #3E2723 (黑褐) | 1.5px / 3px | 细长的山间小路 |
    | **栈道 (plank)** | #8D6E63 (深褐) | #3E2723 (黑褐) | 3px (虚线) | **枕木风格** (实线底+虚线顶) |

### 4. Dijkstra 寻路全面失败 (Pathfinding Always Fails)
- **现象**: 所有道路寻路返回空路径，军队走直线，即使道路数据和图连接完全正确。
- **原因**: `RoadRegistry.ts` 中 Dijkstra 算法使用 `||` 运算符获取距离值：
  ```typescript
  // ❌ BUG: 0 || Infinity → Infinity (JS中0是falsy值！)
  const newDist = (dist.get(current) || Infinity) + edge.weight;
  ```
  起点距离为 `0`，但 `0 || Infinity` 返回 `Infinity`，导致所有邻居距离变成 `Infinity`。
- **解决方案**: 使用 `??`（空值合并运算符）替代 `||`：
  ```typescript
  // ✅ FIX: 0 ?? Infinity → 0（只有null/undefined才回退）
  const newDist = (dist.get(current) ?? Infinity) + edge.weight;
  ```

### 5. 军队生成点偏移 (Army Spawn Offset)
- **现象**: 军队不在城市位置生成，而是在城市周围的偏移点生成。
- **原因**: 旧逻辑曾尝试在城市周围寻找空闲格子。
- **解决方案**: 已修复。现在军队直接在城市坐标（道路起点）生成，支持无限堆叠，确保与路网完美衔接。

## 🔧 几何处理管线 (Geometry Pipeline)

为了解决路网拼接处的**折角(Backtracks)**和**锯齿(Jaggies)**，系统引入了自动修正管线：

```typescript
// src/utils/GeometryUtils.ts
smoothRoad(points) = chaikinSmooth(removeBacktracks(points))
```

1. **去折角 (Remove Backtracks)**: 自动检测并剔除 >100° 的急转弯点（通常是路网拼接 artifacts）。
2. **Chaikin 平滑**: 对剩余路径进行 3 次迭代的切角平滑，生成优美的曲线。
3. **视觉/逻辑同步**: 渲染层 (`Renderer`) 和 逻辑层 (`RoadRegistry`) 共享同一套平滑后的坐标，确保军队行军路线与视觉看到的道路完全一致。

## 🏰 围城定位系统 (Siege Positioning)

军队到达目标城市后，需要在城外道路上驻扎。系统采用**路径回溯 (Path Backtracking)** 算法，确保军队精确停留在道路上。

### 数据流

```
LegionManager.moveLegionToCity()
  ├─ roadRegistry.getFullPathToCity() → [Start, ..., N, City]
  ├─ army.lastPath = [...path]          ← 存储完整路径（含起终点）
  └─ army.moveAlongPath(path.slice(1,-1))  ← 实际行军（去掉起终点）

SiegeManager.positionSiegeArmy()
  ├─ 反转 lastPath → [City, N, ..., Start]
  ├─ 沿道路逐段累计距离，到达 baseDist(0.15°≈16km) 处插值
  └─ 设置位置 + 朝向（Navigation Bearing: atan2(dLng, dLat)）
```

### 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `baseDist` | 0.15 (~16km) | 首个军队与城市的基准距离 |
| `stepDist` | 0.08 (~9km) | 排队军队之间的间隔 |

### 朝向系统

`GlobalUnitRenderer` 使用 **Navigation Bearing** 坐标系：
- `0` = 北 (North)
- `π/2` = 东 (East)
- `π` = 南 (South)
- `-π/2` = 西 (West)

计算公式: `angle = Math.atan2(dLng, dLat)`

> [!CAUTION]
> **不要**使用 Cartesian 坐标系 `atan2(dLat, dLng)` 设置 `lastDirection`！
> 两者差 90°，会导致军队朝向错误。

### 6. 围城军队偏离道路 (Siege Army Off-Road)
- **现象**: 围城时军队不在道路上，而是出现在山脉或其他地形中。
- **原因**: `positionSiegeArmy` 使用直线向量（City + direction × dist）定位，忽略了道路的弯曲。
- **解决方案**: **路径回溯 (Path Backtracking)**
    - `LegionManager` 在派遣军队时，将完整路径存入 `army.lastPath`。
    - `SiegeManager` 在军队到达后，反转路径，沿道路线段逐段回溯，精确插值到目标距离处。
    - 确保军队 100% 停留在道路曲线上。
- **涉及文件**:
    - `Army.ts`: `lastPath` 属性声明
    - `LegionManager.ts`: `moveLegionToCity()` 存储路径
    - `SiegeManager.ts`: `positionSiegeArmy()` 回溯算法
