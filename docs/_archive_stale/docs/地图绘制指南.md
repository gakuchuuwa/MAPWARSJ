# 地图系统规划设计与技术实现文档

本文档详细记录了项目中地图、地区、区域、城市贴图及势力色的展现逻辑及其对应的代码实现文件。

## 1. 核心地图渲染 (Core Rendering)

地图的基础渲染架构，采用 Leaflet 引擎，目前针对“纯地理/历史模式”进行了深度定制。

*   **对应文件**: `src/map/GameMap.ts`
*   **关键功能**:
    *   **底图模式**: 废弃了传统的在线瓦片（Google/OSM），目前核心使用的是 **Procedural Mode**（纯程序化模式）。
    *   **地形渲染**: 依赖 `HillshadeLayer.ts` (Z-Index: 2)，使用高程数据实时渲染山体阴影，提供历史地图的厚重感。
    *   **河流系统**:
        *   **矢量河流**: `VectorRiverLayer.ts` (Z-Index: 335)，用于 Zoom 9 的精细显示。
        *   **ESRI 水系**: `RiverOverlayLayer.ts` (Z-Index: 340)，用于宏观视图。
    *   **可见性控制**: 统一管理图层的 Zoom 级别显示逻辑（如 Zoom 9 特殊处理）。
    *   **城市缩放 (City Scaling)**: [2026-02-12] 采用线性缩放逻辑。
        *   Zoom 9: 1.0x (原始大小)
        *   Zoom 8: 0.5x
        *   Zoom 7: 0.0x (隐藏)
        *   Zoom > 9: 每级增加 0.5x
    *   **山体光照 (Hillshade Lighting)**: [2026-02-12] 采用恒定参数，不再随缩放变化。
        *   Altitude (太阳高度角): **45°** (黄金角度，平衡立体感与遮挡)
        *   Opacity (阴影浓度): **100%** (1.0)

## 12. 🚫 虚拟地形，性能优化 (Performance Optimization) - [2026-01-24 Update]

### 优先队列与时间分片 (Priority Queue & Time Slicing)

为了彻底解决大规模领土计算（如攻陷大城）导致的 **1000ms+ 主线程卡顿**，`TerritorySystem` 实施了两项核心优化：

#### 1. 全局 Dijkstra 算法 (Global Dijkstra)
*   **旧方案**: 轮询式 BFS (`Round-Robin BFS`)。每轮循环都要遍历所有 300+ 个候选城市，寻找当前成本最低的扩展点。复杂度高达 `O(Hexes * Cities)`。
*   **新方案**: **基于优先队列的 Dijkstra**。
    *   引入了 `src/utils/PriorityQueue.ts` (Min-Heap)。
    *   所有城市的扩张前沿（Frontier）统一放入一个全局堆中。
    *   每次仅需 O(1) 取出全局最优解，将复杂度降低至 `O(Hexes * log(Frontier))`。

#### 2. 严格时间分片 (Strict Time Slicing)
*   **问题**: 即使算法优化了，一次性计算数千个格子的归属权仍可能耗时 50-100ms，导致掉帧。
*   **解决**: 引入 `performance.now()` 精确计时。
    *   在 Dijkstra 的 `while` 循环内部，每一步都检查当前帧耗时。
    *   一旦超过 **12ms** (预留 4ms 给渲染)，立即 `ensure requestAnimationFrame` 挂起当前任务，将剩余计算推迟到下一帧。
*   **效果**: 领土变更不再是“瞬间卡顿后刷新”，而是平滑地在 2-3 帧内“流式”展开，彻底消除了长帧卡顿。

### 核心代码
- 领土算法：`src/systems/TerritorySystem.ts` -> `update()`
- 优先队列：`src/utils/PriorityQueue.ts`

## 11. 🌏 地形识别技术 (Terrain Identification Tech) - [2026-01-23 Update]

### 极速全图扫描 (Tile-First Architecture)

为了自动识别游戏中的陆地与海洋（用于决定单位通行权），系统实施了一套基于 **"Tile-First"（瓦片优先）** 架构的扫描方案，成功将 100,000+ 六边形的识别时间从 5分钟+ 降低至 **20秒**。

#### 1. 核心挑战

传统的扫描方法是 **Hex-Centric**（六边形中心）：
1. 遍历每个六边形（10万+）。
2. 计算其中心经纬度。
3. 发起一次网络请求（或调用 `getImageData`）获取该点海拔。
4. **问题**：即使有缓存，10万次异步 Promise 调用和上下文切换也会导致巨大的 CPU 开销和 Event Loop 阻塞。

#### 2. 解决方案：Tile-First 架构

新架构将流程反转：

1. **预计算瓦片需求 (Pre-calculation)**:
   - 首先遍历所有目标六边形，计算出它们总共覆盖了哪些唯一瓦片（Unique Tiles）。
   - 在 Zoom 7 下，全球范围仅需约 1000-2000 张瓦片。

2. **批量并行下载 (Batch Prefetch)**:
   - 使用 `Promise.all` 并发下载所有唯一瓦片（并发数 50）。
   - 将瓦片绘制到离屏 Canvas 并提取 `ImageData` 存入内存 Map。
   - **优势**：网络请求次数从 100,000 次降低到 1,000 次。

3. **同步内存查表 (Synchronous Lookup)**:
   - 瓦片就绪后，再次遍历所有六边形。
   - 此时所有数据均在内存中，直接通过数组索引读取像素值（`imageData.data[idx]`）。
   - **优势**：消除了所有异步等待，处理速度达到微秒级。

#### 3. 数据源与算法

*   **数据源**: Amazon S3 Terrarium Elevation Tiles
    *   URL: `https://s3.amazonaws.com/elevation-tiles-prod/terrarium/{z}/{x}/{y}.png`
*   **海拔解码**:
    *   公式: `Elevation = (R * 256 + G + B / 256) - 32768`
*   **判定规则**:
    *   `Elevation > 0`: **陆地 (LAND)** -> 绿色
    *   `Elevation <= 0`: **海洋 (OCEAN)** -> 深蓝色
    *   **范围外处理**: 用户指定的矩形范围（Lat 0-60, Lng -19-146）之外的所有六边形，无论海拔如何，一律强制标记为 **海洋**（不可通行）。

#### 4. 性能数据

| 指标 | 旧方案 (Hex-Centric) | 新方案 (Tile-First) | 提升 |
| :--- | :--- | :--- | :--- |
| **网络请求** | 100,000+ (逻辑上) | ~1,500 (物理上) | **~60x** |
| **总耗时** | 300秒+ | **20-30秒** | **~10x** |
| **内存开销** | 高 (大量Promise对象) | 中 (ImageData缓存) | 稳定 |

## 3. 城市贴图管理 (City Textures)

城市的外观由“地区”和“规模”共同决定，同时也支持特定名城的独立替换。

*   **对应文件**: `src/systems/RegionSystem.ts` (定义路径), `src/core/CityAssetManager.ts` (旗帜混合)
*   **资源路径映射**: 在 `RegionSystem.ts` 的 `STYLE_MAP` 常量中定义。
    *   **命名规范**: `/cities/[region]_[size].png`
    *   **示例**:
        *   中原大城 -> `/cities/central_big.png`
        *   江南小城 -> `/cities/south_small.png`
        *   塞外关隘 -> `/cities/nomadic_pass.png`
*   **名城特例 (Hardcoded Overrides)**:
    *   在 `getCityImage()` 函数中直接指定。
    *   例如：`changan` (长安) -> `/cities/zhiding/changan.png`
*   **动态旗帜**:
    *   `CityAssetManager.ts` 负责预加载和处理势力旗帜。
    *   支持 **Chroma Key (绿幕抠图)** 技术，动态去除旗帜素材的背景色，使其能自然融合到城市上方。

### 3.1 城市定点与对齐 (City Anchoring & Alignment)

城市图片在六边形地块中的对齐是一个视觉关键点。由于素材尺寸不一（塔楼高耸、城墙扁平），我们采用了特殊的对齐策略：

1.  **Flexbox 容器**:
    *   每个城市图标是一个 `div` 容器，使用 CSS `display: flex; justify-content: center; align-items: center;`。
    *   这确保了无论图片本身的宽高比如何，它在容器内部始终居中。

2.  **几何中心锚点 (Geometric Center Anchor)**:
    *   Leaflet 的 `iconAnchor` 被动态计算为 `[size/2, size/2]`。
    *   通过将锚点设置在容器的**严格几何中心**位置，我们确保了城市图标的中心点与六边形的地理中心完全重合。
    *   此修改源于 2026-02-08 的**所有实体统一中心对齐原则**，废弃了旧版为了模拟所谓“站立透视”而进行的垂直偏移。

### 3.2 交互与坐标系统 (Interaction & Coordinate System)

在实现“城市点击”与“地图格子”对齐时，遇到的核心技术挑战及解决方案：

1.  **Leaflet Marker 点击偏移问题**:
    *   **现象**: Leaflet 的 `marker.on('click', e)` 事件中，`e.latlng` 默认返回的是 Marker 的**锚点位置** (Anchor Position)，**而非**鼠标点击的屏幕位置。这导致对于大尺寸图标（如城市），用户点击图标顶部（视觉上的格子）时，系统判定的却是图标底部锚点所在的格子。
    *   **解决方案**: **不可信任 `e.latlng`**。必须使用 `map.mouseEventToLatLng(e.originalEvent)` 从原生的 DOM MouseEvent 获取精确的屏幕坐标，再转换为地理坐标。实现“所点即所得”。

2.  **视觉网格 vs 逻辑网格**:
    *   城市图标具有高度（2.5D 透视），其顶部可能跨越到北方的格子上。
    *   通过上述的精确坐标转换，结合 Hex 网格计算，确保了用户点击图标的哪部分，就选中该部分下方的格子，消除了视觉与逻辑的错位感。

3.  **穿透交互 (Click-Through)**:
    *   在特定模式下（如绘路模式），为了防止城市图标阻挡下方的路网编辑，我们动态设置了城市 Pane 及其所有子元素的 CSS `pointer-events: none !important`。
    *   **注意**: 仅设置 Pane 的样式是不够的，必须遍历设置所有 Marker DOM 元素，因为 Leaflet Marker 默认通过 JS 绑定了强交互事件。

## 4. 区域/疆域系统 (Territory System) - 势力范围

“区域”是指游戏内的领土控制权，表现为地图上的色块覆盖。

*   **对应文件**: `src/systems/TerritorySystem.ts`
*   **计算逻辑**:
    *   **算法**: 使用 **Dijkstra** (或 BFS) 算法，以城市为中心向外辐射。
    *   **权重**:
    *   **权重**:
        *   **大城 (Big City)**: 辐射 3 格。
        *   **中城 (Medium City)**: 辐射 2 格。
        *   **小城/关隘/渡口**: 辐射 1 格。
        *   **道路**: 沿道路扩张消耗极低（0.05），可延伸很远。
        *   **地形**: 离开道路进入荒野消耗极高（1.0），且受最大物理半径限制。
*   **渲染表现**:
    *   **色块**: 使用 `L.polygon` 绘制。
    *   **动态效果**:
        *   Zoom <= 8: 实体色块填充（战略视图）。
        *   Zoom 9-10: **光晕边界 (Glow Border)** 模式，仅显示边缘发光带。
        *   Zoom >= 11: 隐藏色块，只看地形细节。

### 3.5 长城渲染风格 (Great Wall Rendering)

针对秦汉时期的**夯土长城**特性，我们放弃了传统的“铁轨式”符号，转而采用**3D 拟真与手绘质感结合**的渲染方案：

1.  **3D 分层结构 (Sandwich Layering)**
    为了体现土埂的隆起感，我们在 `GameMap.ts` 中创建了三个专用 Pane，并通过 CSS 偏移实现立体视差：
    *   **阴影层 (Shadow Layer)**: 底层，黑色半透明，向右下方偏移 (+3px)，模拟长城投下的阴影。
    *   **本体层 (Base Layer)**: 中层，赭石色 (#A0825A)，代表夯土墙体。
    *   **高光层 (Highlight Layer)**: 顶层，浅米色 (#DCC8A0)，向左上方偏移 (-2px)，模拟受光面。

2.  **手绘抖动效果 (Organic Jitter)**
    *   **技术**: 使用 SVG Filter (`feTurbulence` + `feDisplacementMap`)。
    *   **效果**: 对规整的矢量线条进行微小的随机扰动，模拟夯土墙随地形起伏、经风雨侵蚀后的不规则边缘，避免现代工业产品的“光滑感”。
    *   **实现**: 通过 `VectorWallLayer.ts` 动态向 DOM 注入 SVG Filter 定义，并应用到所有墙体 Path 上。

3.  **烽火台 (Tower Icons)**
    *   **形态**: SVG 绘制的**梯形**图标（上窄下宽），而非正方形，更符合实物。
    *   **光影**: 内置了深色底座、土色本体、亮色高光三层结构，与墙体光影保持一致。
    *   **布局**: 仅在关键节点生成 (Step=8)，避免过度密集。
    *   **层级**: 渲染在独立的 `vectorWallNodePane` (Unknown Z-index, Topmost)，确保永远覆盖在墙体线条之上。

### 7.2 矢量古道 (Vector Roads) - **Legacy / Reference**

为了还原“古道”在复杂地形（如秦岭、子午岭）中蜿蜒穿行的真实感，系统引入了脱离六边形网格限制的**矢量道路**层。

*   **对应文件**:
    *   **渲染层**: `src/map/VectorRoadLayer.ts`
    *   **数据源**: `src/data/VectorRoadData.ts` (GeoJSON 格式)
*   **渲染技术**: **双层描边 (Dual-Layer Casing)**
    *   为了在任何底图（雪山、黄土、深岩）上都能清晰可见，道路采用两层绘制：
    *   **底层 (Border)**: 深褐色 (`#4E342E`) 加粗线条，负责与浅色背景形成高对比轮廓。
    *   **顶层 (Inner)**: 朱砂红 (`#D84315`, 类似秦直道夯土/官道红) 亮色细线，负责在深色背景中发光显形。
*   **AI 寻路集成 (Pathfinding Integration)**:
    *   **原理**: 虽然视觉上是连续曲线，但逻辑上通过 `GridSystem.latLngToAxial` 将矢量点映射为**虚拟网格路网**。
    *   **实现**: 在 AI 寻路 (`HexPathFinder`) 计算代价时，检索“虚拟路网注册表”。凡是处于矢量路径上的格子，即使没有玩家手绘的“黄格子”，也会被强制视为 `Cost = 1.0` (畅通)，从而引导 AI 沿着视觉上的古道行军。

常见问题 (Troubleshooting) - [2026-01-27 Update]

### 道路不可见问题 (Invisible Roads)
如果您在 `src/data/RoadData.ts` 中添加了新的道路坐标，刷新页面后却无法显示，请检查以下两点：

1.  **数据位置错误 (Wrong Array)**:
    *   **现象**: 坐标数据位于文件**最末尾**。
    *   **原因**: `RoadData.ts` 文件末尾定义了一个 `DISABLED_ROADS_DATA`（禁用道路）数组。如果您直接在文件底部的 `];` 前插入数据，实际上是将道路加入了禁用列表。
    *   **解决**: 请确保将新数据添加到文件上方的 **`CUSTOM_ROADS_DATA`** 数组中（约第 3300 行附近）。

2.  **区域纹理缺失 (Missing Texture)**:
    *   确保该区域（如 `NOMADIC` / `SIBERIA`）在 `RoadLayer.ts` 中有对应的纹理映射。

## 8. 🗺️ 领土系统 (Territory System)

### 3级城市分类 (3-Level / 5-Type Classification)

领土范围基于城市类型自动判定，分为 3 个级别、5 种类型：

| 级别 | 类型 | 领土圈数 | 说明 |
| :--- | :--- | :--- | :--- |
| **一级** | **大城 (Big City)** | **3圈** | 首都、区域核心大城 |
| **二级** | **中城 (Medium City)** | **2圈** | 标准城市 |
| **二级** | **关隘 (Pass)** | **2圈** | 军事要塞、关口 |
| **三级** | **小城 (Small City)** | **1圈** | 小型据点 |
| **三级** | **渡口 (Ferry)** | **1圈** | 渡口、码头 |

### 双模式领土扩展 (Dual-Mode Territory Expansion)

系统采用"核心圈 + 道路连接"的双模式扩展逻辑：

1. **核心领土 (Core Territory)**
   - 严格限制在城市中心的物理距离内
   - 使用 BFS 算法计算，每格消耗成本 1.0
   - **硬性物理距离检查**：无论路径成本多低，草地格子超出圈数限制一律不可占领

2. **道路连接 (Road Connectivity)**
   - 道路格子成本极低（0.05），可无限延伸
   - 城市可"无线连接"5格内的最近道路
   - **绝缘机制**：在核心圈外的道路上时，禁止向任何草地扩展，保持道路势力线细长

3. **优先权规则 (Priority Rules)**
   - 物理距离优先于路径成本
   - 第1圈绝对优先 → 第2圈 → 第3圈 → 道路
   - 防止远距离城市通过道路"抢夺"近距离城市的核心领土

- 道路数据：`src/data/RoadData.ts`

## 9. 高精度矢量道路绘制规范 (High-Precision Vector Road Protocol) - [Legacy]

*(注：此部分仅适用于已废弃的矢量道路系统)*

为了实现如 **滏口陉 (Fukou Jing)** 和 **河内道 (Henei Road)** 般精确还原历史与地理逻辑的道路，请遵循以下 SOP：

### 9.1 核心流程 (Workflow)

1.  **历史考据 (Strategic Analysis)**
    *   确定起点、终点及必须经过的**关键据点**（如：关隘、古渡口）。
    *   *案例*: 滏口陉必须经过“响堂山南麓缺口”而非直接翻越鼓山。

2.  **路线拟合 (Terrain Fitting)**
    *   **原则**: 古人行军**走水不走山**。
    *   **工具**: 使用 Google Earth 或地形图，沿着河谷（如清漳河、沁河）或山麓等高线描点。
    *   **精度要求**:
        *   平原段: 每 10-20km 一个坐标点。
        *   山区段: 关键转弯处（如 V 型隘口）需加密至 1-2km 一个点，以形成平滑曲线。
        *   **保留 4 位小数** (约 10 米精度)。

3.  **逻辑收束 (Logical Convergence)**
    *   **多路会师**: 当多条道路汇入同一大城（如邯郸）时，**不要**让它们各自直接连向城市中心。
    *   **汇合点 (Convergence Point)**: 应在城外 5-10km 处设置一个公共坐标点，让多条道路在此合并，形成“大军集结，同路入城”的视觉压迫感。

### 9.2 数据录入标准

在 `src/data/VectorRoadData.ts` 中录入时：
*   **ID**: 必须唯一且具语义 (e.g., `path_fukou_jing`).
*   **Color**:
    *   普通古道: 默认。
    *   特殊走廊: 如河内道使用 `#8B4513` (深棕) 以示区分。
*   **Coordinates**: 严格按照 `[lng, lat]` (经度, 纬度) 格式录入 GeoJSON。

### 9.3 验证标准
1.  **视觉验证**: 缩放至 Zoom 10，检查道路是否“切山”（错误）或“沿河”（正确）。
2.  **逻辑验证**: AI 军队是否能顺畅地从起点沿此路移动到终点（依赖 `RoadRegistry` 的自动插值算法）。


## 7. 🎨 视觉技术文档

### 动态 SVG 领土光晕 (Dynamic SVG Territory Glow)

为了在保持高性能的同时实现柔和的势力边界，`TerritorySystem` 采用了混合渲染技术：

1.  **几何合并**: 使用算法将相邻的同势力六边形合并为单一的多边形路径，极大减少了 DOM 节点数量。
2.  **SVG 滤镜**: 利用 SVG `<filter>` 实现高性能的内/外发光效果（Inner/Outer Glow），替代了昂贵的 Canvas 逐像素模糊计算。
    -   这允许在地图缩放时保持边界的平滑与光晕的稳定，同时利用 GPU 加速渲染。

### 内向型光晕彩带 (Inward Glow Ribbons) - Zoom 9

为了在宏观战略视角 (Zoom 9) 实现"仅在国家交界处向内发光"的各种苛刻视觉要求，系统采用了一套混合几何算法：

1.  **Shared Border Detection**: 在 `renderFactionBorders` 中，系统会逐段检测六边形边界。仅当邻居 Hex 属于不同势力（且非空地）时，该边界被标记为"Shared Border"。
2.  **Ribbon Geometry Generation**: 针对每一段 Shared Border，算法会自动计算指向六边形中心的向量，生成一个向内延伸 20%-25% 的四边形带 (Ribbon Quad)。
3.  **Layering & Blur**:
    - 这些 Ribbon Quads 被合并渲染在黑色国界线图层**下方**。
    - 通过 CSS `.glow-ribbon` 类应用高斯模糊 (`filter: blur(4px)`)。
    - 结果：黑色边界线保持锐利，而光晕仅在边界内侧柔和晕开，且物理上严格限制在领土内部，绝无外溢。

## 10. 实体对齐标准 (Map Entity Alignment Standard) - [2026-02-08]

为了保证**俯视地形图 (Top-down Terrain Map)** 风格的统一性，系统摒弃了一切“伪3D透视”带来的垂直偏移。所有地图实体必须遵循**几何中心严格对齐**原则。

### 10.1 统一规范 (Unified Protocol)

*   **原则**: 任何可放置在六边形上的实体（城市、军团、路点），其渲染的**几何中心**必须严格重合于六边形的**经纬度中心**。
*   **禁止**: 
    *   禁止使用“脚底对齐” (Bottom Anchor) 模拟站立感。
    *   禁止为特定兵种或建筑添加额外的垂直偏移 (Y-Offset)。

### 10.2 具体实现 (Implementation)

1.  **城市 (City)**
    *   **文件**: `src/systems/TerritorySystem.ts`
    *   **Anchor**: `[0.5, 0.5]` (Center)
    *   **逻辑**: 图标容器也是 Flex 居中，无任何 Padding 偏移。

2.  **军团 (Legion)**
    *   **文件**: `src/map/legion/LegionPhalanxDrawer.ts`
    *   **Anchor**: `0.5` (Center)
    *   **说明**: 军团图标绘制在六边形正上方。虽然野战时会有左右偏移以区分敌我，但在垂直方向上始终居中。

3.  **道路 (Road)**
    *   **文件**: `src/map/RoadLayer.ts`
    *   **逻辑**: 连接两个六边形的中心点。纹理通过 Canvas 绘制在连线中点，无偏移。
