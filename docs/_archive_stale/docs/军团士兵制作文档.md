## 2. 技术方案

### 2.1 绿幕抠图 (Chroma Key)
**问题**：原始素材带有 `#00FF00` (或相近绿色) 的背景。
**解决**：在 `PhalanxDrawer` 预加载阶段进行像素处理。
- 使用 `OffscreenCanvas` 或临时 Canvas 加载图片。
- 获取 `ImageData`。
- 遍历所有像素，检测绿色分量。
  - 判定标准：`G > 200` 且 `R < 100` 且 `B < 100` (容差可调)。
  - 处理：将符合条件的像素 Alpha 值设为 0 (完全透明)。
- 将处理后的数据转回 `HTMLImageElement` 或直接存储为 `ImageBitmap` 供渲染使用。

### 2.2 动画切片 (Sprite Strip)
**问题**：素材是长条图，包含8个连续动作。
**解决**：
- **自动检测**：如果 `图片宽度 / 图片高度 > 4`，判定为动画序列帧。
- **帧数计算**：`帧数 = Math.floor(宽度 / 高度)` (通常为 8)。
- **渲染逻辑**：
  - 除了现有的 `direction` (8方向) 外，引入 `tick` (时间戳) 参数。
  - `当前帧 = (Math.floor(tick / 帧时长) % 总帧数)`。
  - 使用 `ctx.drawImage` 的源矩形参数 `sx, sy, sWidth, sHeight` 仅绘制当前帧。

*   **核心逻辑**: **离锚点 (Anchor) 最近的士兵即为前排**。

### 4.5 死亡表现
*   **逻辑**: 兵力减少导致 `visibleCount` 下降时，触发阵亡。
    *   **坐标锁定**: 尸体渲染时会**锁定在世界坐标 (Lat/Lng)**，不随军团移动而移动。
    *   **清理机制**: 
        *   战斗中：尸体保留。
        *   战斗结束 (`isFighting = false`)：自动清理所有尸体。
        *   初始化防止闪烁：初始化时仅创建 `visibleCount` 个单位，防止多余单位瞬间死亡导致尸体“诈尸”。
*   **动画**: 取消专门的“受伤(DYING)”动画，直接播放“阵亡(DEAD)”动画然后消失。

## 6. 方阵活力化系统 (Phalanx Vitality System) [New]
为解决方阵“僵硬”问题，引入了以下微观动画系统（仅应用于玩家和正规军团）：

### 6.1 错落动画 (Staggered Animation)
*   **问题**: 所有士兵动作完全同步（同时挥刀）。
*   **解决**: 基于士兵的 Grid 坐标生成随机种子，对动画帧索引 (`frameIndex`) 施加随机偏移。
*   **效果**: 即使是同一个兵种，不同位置的士兵动作也会有自然的时间差。

### 6.2 阵型波浪 (Gentle Wave)
*   **问题**: 士兵站位如钉死般固定。
*   **解决**: 在 `ATTACK` 状态下，引入低频的正弦波 (`Math.sin`) 偏移每个人物的 X/Y 坐标。
*   **弃用**: 曾尝试高频抖动 (Jitter)，因效果像帕金森而被移除。
*   **效果**: 模拟大军如麦浪般轻微起伏。

### 6.3 战斗呼吸 (Combat Breathing)
*   **问题**: 整体阵型是一个死板的矩形。
*   **解决**: 在 `ATTACK` 状态下，随时间调整 `spacingX` 和 `spacingY` (间距系数)。
*   **参数**: 周期约 2.5秒 (400 tick factor)，幅度 1% (0.01)。
*   **效果**: 整个方阵会随着战斗节奏有极微妙的“舒张”感。


### 6.4 动态前排 (Dynamic Frontline)
*   **逻辑**: 严格区分前排与后排。
    *   **前排 (Frontline)**: 执行 **攻击 (Attack)** 动作，承担战斗表现。
    *   **后排 (Backline)**: 执行 **待命 (Idle)** 动作，保持阵型严整，不再随乱挥刀。
*   **目的**: 增强战术纪律感，视觉上突显“交锋线”。

## 7. 常见坑点与注意事项 (Critical Pitfalls) [IMPORTANT]

### 7.1 方向映射错误 (Direction Mapping Error)
**此错误已发生多次，请务必注意！**
代码中的 `direction` (0-7) 与视觉上的 Frontline 判定存在特殊的等轴侧映射关系，**绝对不是**简单的罗盘对应。

*   **错误认知**: "朝西(West)" -> Code 6 -> Check Left? **WRONG!**
*   **正确映射**:
    *   **Code 0 (Visual NE)**: 朝向右上。前排在 Top & Right。
    *   **Code 1 (Visual E)** : 朝向右方。前排在 Right (Col+1)。
    *   **Code 2 (Visual SE)**: 朝向右下。前排在 Bottom & Right。
    *   **Code 3 (Visual S)** : 朝向下方。前排在 Bottom (Row+1)。
    *   **Code 4 (Visual SW)**: 朝向左下。前排在 Bottom & Left。
    *   **Code 5 (Visual W)** : **朝向左方**。前排在 **Left (Col-1)**。
    *   **Code 6 (Visual NW)**: 朝向左上。前排在 Top & Left。
    *   **Code 7 (Visual N)** : 朝向上方。前排在 Top (Row-1)。

**判断逻辑**: 必须使用 `PhalanxState.ts` 中的 `Visual Mapping` 表，严禁使用标准几何推导。

### 7.2 对象区分 (Entity Distinction: Player vs Legion) [CRITICAL]
**严禁混淆玩家与NPC军团！**
游戏中有两类独立的战斗单位，它们共享部分渲染逻辑，但配置完全独立：

1.  **玩家 (Player)**:
    *   **渲染器**: `LegionRenderer.ts`
    *   **贴图**: `SPRITE_PATHS.PHALANX` (Player Set)
    *   **尺寸**: **100** (Standard)
    *   **控制**: 玩家直接控制。

2.  **正规军团 (AI Legion)**:
    *   **渲染器**: `GlobalUnitRenderer.ts`
    *   **贴图**: `SPRITE_PATHS.AI_PHALANX` (Legion Set)
    *   **尺寸**: **50** (Small)
    *   **控制**: AI 逻辑控制。

**修改原则**: 修改其中一个时，**必须**检查是否意外影响了另一个。二者在 `PhalanxDrawer.ts` 中通过 `variant` 参数 ('player' | 'legion') 严格区分。

## 8. 技术实现总结 (Technical Summary) [Success]

### 8.1 核心架构 (Core Architecture)
*   **状态分离 (State Separation)**:
    *   **旧逻辑**: 渲染与逻辑混杂在 `PhalanxDrawer` 中，导致状态难以追踪。
    *   **新方案**: 引入 `PhalanxState.ts` 独立管理每个军团的 `UnitState` (Rows, Cols, Strings, States)。`PhalanxDrawer` 退化为纯渲染器。
*   **战斗锁定 (Combat Locking)**:
    *   进入战斗瞬间，记录当前的 `maxRows`, `maxCols` 和 `maxTroops` (作为 100%均值)。
    *   战斗中禁止阵型重排 (Resizing)，确保死尸位置固定，阵型只有减少没有变形。

### 8.2 关键算法解决方案 (Key Solutions)

#### A. 瞬死问题 (Instant Wipe-out)
*   **现象**: 兵力减少少量 (如 50人)，导致视觉上所有小人 (如 100个) 全部阵亡。
*   **原因**: 错误地将“兵力数值损失”直接映射为“视觉单位数量损失”。(50 兵力 > 100 视觉单位? No, but logic was flawed).
*   **修复**: **比例映射法 (Ratio Mapping)**。
    *   公式: `TargetAliveCount = ceil(TotalSlots * (CurrentTroops / InitialMaxTroops))`。
    *   效果: 兵力损失 10%，视觉单位也只倒下 10%。

#### B. 前排识别错误 (Wrong Frontline)
*   **现象**: 敌人从西边 (左侧) 进攻，甚至背后，前排士兵却在错误的位置挥刀。
*   **原因**: 混淆了 **Code Direction (0-7)** 与 **Screen Visual Direction**。在等轴侧视角下，Code 5 (通常为 NE) 实际上对应屏幕的“左方 (West)”。
*   **修复**: **视觉射线检测 (Visual Raycast)**。
    *   建立严格的映射表 (详见 7.1)。
    *   只判定“视觉上的前方邻居”是否存活。若前方邻居已死或出界，则我就是前排。

### 8.3 视觉优化
*   **随机倒下 (Random Death)**: 士兵阵亡时不再统一朝向，而是随机 `0-7` 方向倒下，增加战场的混乱与真实感。
*   **非战斗呼吸 (Idle Breathing)**: 仅在战斗且为前排时播放攻击动画，后排待机移动，层次感极强。

四，出兵系统。

### 1. 触发条件 (Trigger Condition)
- **单军团原则**: 每个阵营同一时间只能有一支主力军团在地图上活动。已有军团存活时，不会生成新军团。
- **兵力阈值**: 阵营所有城市的**可用兵力**（当前兵力 - 最低守军）总和必须大于 **20,000** 人。
  - *目的：防止兵力不足时频繁送死 (添油战术)。*
- **冷却时间**: 
  - 出兵成功：冷却 **60秒**。
  - 出兵失败（如无兵可调）：冷却 **30秒**。

### 2. 出兵兵力 (Troops)
- **数量设定**: 不设上限 (`Infinity`)。
- **实际执行**: 系统会征召阵营内所有城市的**盈余兵力**（保留最低守军后），有多少出多少。

### 3. 出兵地点 (Spawn Location)
- **优先级**: 
  1. 首都 (Capital)
  2. 皇都/巨城 (Imperial/Huge City)
  3. 大城 (Large City)
  4. 中城 (Medium City)
- **限制**: 只有上述类型的城市可以出兵。**关隘 (Pass) 和 小城 (Small City) 无法出兵**。

### 4. 目标选择 (Target Selection)
- **逻辑**: 寻找距离**最近**的敌方据点。
- **范围**: 包含所有**敌方城市、关隘 (Pass) 和 要塞 (Fortress)**。

五，战斗系统。

### 1. 核心机制
   
- **物理化时长**: 战斗时长由**总兵力规模**决定，而非瞬间结算。
  - 公式: $\text{Duration} = \text{TotalTroops} / 1000$
  - 范围: 5秒 ~ 45秒。


### 2. 战力计算

- **基础战力**: `兵力 (Troops)`。
- **运气成分**: 每次战斗开始时，双方（**无论是军队还是城市守军**）都会独立获得一个 **0.5x ~ 1.5x** 的随机战力倍率。
- **以少胜多**: 现在的方差非常大！理论上，运气极好的弱势方（1.5x）甚至可以战胜兵力是自己 **3倍** 但运气极差（0.5x）的对手。(1.5 / 0.5 = 3.0)
- **用范围**: 野战 (Army vs Army) 和 攻城战 (Army vs City) **通用**。守城方即使是静态单位，也拥有同样的“天命”判定。
- **关隘特权 (Pass Bonus)**: 
  - **防守修正**: 关隘 (Pass) 拥有 **0.5x** 的伤害减免系数（即受到的伤害只有正常的 50%）。
  - **战略意义**: 攻打关隘需要 2 倍于平常的兵力或时间。
- **基础战力**: 完全等于**兵力 (Troops)**。
- **适用范围**: 所有战斗通用。

### 3. 战斗结果
- **伤兵机制**: 获胜方在战斗结束后，立即恢复 **30%** 的阵亡兵力。
- **输家惩罚**: 
  - 野战：溃灭或撤退。
  - 攻城：防守方兵力归零，城市易主（重置为 1000 兵）。


本文档详细说明了当前版本（2026-01-24）的战斗系统设定与逻辑实现。

## 1. 核心设计理念

*   **独立实体**：军团（Legion/Army）与城市（City）是完全独立的实体。军团在地图上移动，城市固定不动。军团无法"进入"城市，只能驻扎在城市附近。
*   **物理化节奏**：战斗不是瞬间结算，而是基于"兵力规模"计算持续时间（5秒 - 20秒），模拟真实的交战过程。
*   **简化数值**：战斗力（Power）直接对应兵力（Troops），大兵团优势显著。
*   **1v1原则**：为了避免逻辑混乱，同一地块/同一单位在同一时间只能参与一场战斗。第三方试图介入时会被"阻塞"（Blocked）。

## 2. 触发机制 (Contact Engine)

战斗主要通过 `ContactEngine` 和 `SiegeManager` 触发。

### 2.1 野战 (Field Battle)
*   **触发条件**：当军团移动到敌对军团的**相邻六边形**（Adjacent Hex）时触发。
*   **判定逻辑**：
    1.  检测移动后周围 6 个格子的单位。
    2.  如果是敌对阵营 (Faction ID 不同)。
    3.  如果目标正处于战斗中 -> **动态增援** (检查是否可加入当前战场，若可加入则变为援军，否则阻塞)。
    4.  如果目标空闲 -> **触发野战**。
*   **结算后**：胜利方会有 2 秒的"冷却/僵直"时间，期间保持战斗状态，防止连续极其快速的接敌。

### 2.2 攻城战 (Siege Battle)
*   **触发条件**：
    1.  **碰撞触发**：军团试图移动到敌对城市所在的六边形（碰撞）。
    2.  **区域拦截 (ZOC)**：军团移动到敌对城市相邻格子时，会被城市拦截并强制进入攻城战。
    3.  **事件触发**：通过剧情/AI指令直接发起的攻城。
*   **判定逻辑**：
    *   如果城市是友方 -> 停止移动（驻驻）。
    *   如果城市正在被围攻 -> **动态增援** (若是相同阵营则加入攻/守方，若是第三方则阻塞)。
    *   否则 -> **触发攻城战**。

## 3. 战斗核心机制 (Combat System)

当前采用"时间驱动"（Time-Driven）的战斗模型。

### 3.1 时长计算 (Target Duration)
战斗持续时间并非固定，而是根据双方总兵力动态计算，以模拟大规模战役的胶着感。

$$
\text{主要公式: } \text{Duration} = \text{Base} + (\log_{10}(\text{TotalTroops}) \times \text{MassFactor})
$$

*   **基准时间 (Base)**: 5.0 秒
*   **规模系数 (Mass Factor)**: 2.5
*   **上下限**: 最短 5 秒，最长 30 秒。
*   **示例**:
    *   极小规模 (几百人): ~5 秒
    *   中等规模 (1万人): ~15 秒
    *   大规模 (10万人+): ~17.5 - 30 秒

### 3.2 伤害计算 (DPS Model)
系统预先计算"失败者需要多久死光"，以此反推每秒伤害（DPS）。

1.  **胜负预测**: 比较双方兵力（Troops）。兵力多者为预测胜者。
2.  **输家DPS (Loser DPS)**: $\frac{\text{输家当前兵力}}{\text{目标时长}}$
    *   保证输家在战斗结束时兵力归零。
3.  **赢家DPS (Winner DPS)**: $\text{输家DPS} \times \frac{\text{输家兵力}}{\text{赢家兵力}}$
    *   赢家受到的伤害与双方兵力比成反比。兵力优势越大，战损越小。

### 3.3 修正与保护
*   **保底机制**: 赢家兵力不会低于初始兵力的 10%（防止惨胜导致双方同归于尽）。
### 3.4 战斗结束
*   当任一方兵力低于 1.0 时，战斗结束。
*   **输家**: 兵力归零，状态设为 `isDestroyed`，从地图移除。

*   **例外**: 若赢家方有单位兵力归零（如守城军死光但援军赢了），该单位**免于销毁**，强制恢复至 **1000 兵力**（最低安全阈值），确保依然存活。

## 4. 兵种与军团 (Legions & Units)

### 4.1 兵种属性
目前的兵种差异主要体现在**视觉表现**上，数值上统一使用"兵力"计算。

配置 (`GameConfig.ts`) 定义了不同阵营的兵种外观：
*   **华夏系 (Huaxia)**: 步兵、骑兵、混合部队（步骑弩）。
*   **越南系 (Yuenan)**: 藤甲兵。
*   **回回系 (Huihui)**: 弓骑兵。


### 4.2 军团管理
*   **创建**: 军团只能通过事件或城市动员（Sortie）创建，不存在凭空刷新。
*   **动员**: 从城市征兵出战时，会扣除城市守军。
*   **移动**: 推荐使用道路移动 (`moveLegionToCity`)，系统会优先寻找道路路径。

## 5. 参数配置概览 (关键值)

### 战斗动画技术 (Combat Animation Tech) [2026-01-16 Update]

系统引入了高度风格化的微观战斗动画，以增强宏观推演的生动性：

2.  **动态冲锋 (Dynamic Charge)**:

    *   **相位离散**: 每个骑兵拥有独立的动画相位偏移 (Phase Offset)，避免了机械的同步摆动。

3.  **位置抖动 (Micro Jitter)**:
    *   战斗状态下，所有单位都会叠加高频的随机位置抖动 (+/- 8px)，模拟战场上的混乱与厮杀感。

4.  **视觉对齐与平衡 (Alignment & Balance)**:
    *   **脚底锚点**: 渲染锚点从“膝盖(0.65)”修正为“脚底(0.90)”，确保无论单位缩放比例如何，所有士兵的脚底绝对水平对齐。
    *   **差异化缩放**: 针对不同素材源（如巨大的弩兵与较小的步兵），系统底层应用了 **差异化缩放 (1.2x vs 0.9x)**，确保最终呈现的视觉大小一致。

### 纯正染色技术 (True Color Tinting) [2026-01-16 Update]

为了解决原始素材蓝色底色干扰阵营染色的问题 (如 蓝底+红染=紫色)，渲染层重构了 `SpriteTinter` 算法：

*   **去色底 (Desaturation)**: 算法不再进行简单的 RGB 混合，而是先提取原始像素的 **亮度信息 (Luminance/Grayscale)**。
*   **重着色 (Re-Coloring)**: 基于提取的灰度值与目标阵营色进行混合。
    *   **公式**: `Pixel = Brightness * (1 - Intensity) + TargetColor * Intensity`
    *   **结果**: 彻底消除了素材底色干扰，实现了 **"红即是红 (Red is Red)"** 的纯正色彩表现，同时保留了原始素材的光影立体感。

## 10. 🏰 攻城定位系统 (Siege Positioning System) - [2026-01-22 Update]

### 核心问题与解决方案

在六边形地图中，军队不能与城市重叠在同一个格子里。因此当军队发起攻城战时，需要将其定位到城市的**相邻六边形**上。

#### 原始问题
之前的实现存在"军队乱跳"的 Bug：军队从道路行军过来，攻城战开始后却被移动到完全错误的位置（如对面方向）。

#### 问题根因
经过深入排查，发现问题出在**两处代码同时尝试定位军队**：

1. **`LegionManager.triggerSiege`**：碰撞检测触发攻城时，会尝试将军队移动到相邻格。
2. **`SiegeManager.onArmyArrive`**：攻城逻辑开始时，再次尝试定位军队。

更关键的是，碰撞检测的流程会**先回滚军队位置**：
```
军队尝试进入城市格 → 碰撞失败 → 位置回滚到道路格子 → 触发攻城
```

所以当攻城逻辑运行时，军队**已经在正确的道路格子上**了！但旧代码假设军队在城市格子内，试图把它"移出来"，导致了混乱。

### 修复方案

**核心原则：军队通过道路碰撞触发攻城时，已经在正确的位置上，不需要移动！**

新逻辑（`SiegeManager.onArmyArrive`）：

```typescript
// Step 1: 获取军队当前实际位置
const armyHex = GridSystem.latLngToAxial(army.getPosition());

// Step 2: 检查是否已经在城市的相邻格
const isAlreadyAdjacent = adjacentHexes.some(h => h.q === armyHex.q && h.r === armyHex.r);

if (isAlreadyAdjacent) {
    // ✅ 已在正确位置 → 只需吸附到格子中心，设置朝向
    army.setPosition(hexCenter);
    army.lastDirection = OrientationSystem.get8DirectionIndex(hexCenter, cityPos);
} else {
    // ⚠️ 不在相邻格（异常情况）→ 寻找最近空位并移动
    // ...
}
```

### 配置位置
- 定位逻辑：`src/core/SiegeManager.ts` (第405-470行)
- 碰撞检测：`src/core/LegionManager.ts` (第272-400行)

### 设计要点

1. **单一定位点**：所有攻城定位逻辑统一在 `SiegeManager.onArmyArrive` 处理，`LegionManager.triggerSiege` 不再进行任何位置操作。
2. **状态检查优先**：先判断军队当前状态，而非盲目执行移动。
3. **朝向自动计算**：使用 `OrientationSystem.get8DirectionIndex` 确保军队面向城市。
