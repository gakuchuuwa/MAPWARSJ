# 14 文化区「可玩陆地」与据点比例测算

> **读者**：项目维护者、日后协作 AI。  
> **用途**：审计各文化区据点是否过多/过少；**不**用于改坐标、**不**用于自动画路。  
> **实现脚本**：`scratch/region_playable_area.mjs`  
> **机器可读输出**：`scratch/region_playable_area.json`（每次运行覆盖）

---

## 一、方法论总纲（两层，见好就收）

全图据点约 **652 城**。审计采用 **「数学打底 + 历史定调」**，分两层，**禁止**在代码里对东北 / 河西 / 中亚再叠几何掩膜。

| 层 | 做什么 | 不做什么 |
|----|--------|----------|
| **第一层：绝对禁区硬扣** | 界城环线内，扣塔克拉玛干、羌塘、戈壁大漠、滇缅雨林、日朝海域 | 不扣兴安岭、中亚沙漠、河西以北戈壁（那是渔猎/游牧/走廊 **节点带**） |
| **第二层：局部历史密度** | 东北 / 中亚 / 河西：几何建议 × **0.6**；中原 / 南方 / 北方：参考 × **1.2 / 1.15 / 1.1** | 不做全局重分配（避免中原被抬到 90+ 城） |
| **第三层（人工）** | 文化中心、界城锚点、652 健康版为锚；道路由主人手画（见 no-road-drawing 规则） | 不由脚本自动删城 |

### 为何「减来减去」对边陲反而不好？

若把大兴安岭、中亚沙漠、河西戈壁 **全部几何扣掉**，蒙特卡洛 会给出东北 ~15 城、中亚 ~12 城、河西 ~8 城——边陲本就有 **400–600 km 级** 节点间距，这属常态，**不以汉地驿道长度（如酒泉↔瓜州 ~300 km）作为硬顶**。

- **抹杀文明类型**：女真 / 靺鞨 / 游牧 / 粟特商道，历史上本依托 **山林、沙漠边缘、水源地** 设据点，不需要中原级良田；把地貌全标「不可玩」等于否定这些势力生存空间。

**结论**：第一层只扣 **绝对生命禁区**（无人能在那建城）；边陲「地广人稀」用 **第二层系数** 折中，不用掩膜硬砍；**边陲远距 ≠ 死档**。

### 第一层仍须硬扣的失真（几何）

| 失真来源 | 处理 |
|----------|------|
| 日朝环线含海 | 仅计列岛 / 半岛陆地 |
| 塔克拉玛干、古尔班通古特等 | 西域硬扣 |
| 羌塘、阿里无人区 | 青藏硬扣 |
| 戈壁大漠（非走廊节点） | **仅草原**硬扣 |
| 滇缅湿热雨林 | 硬扣（与坝子/走廊分开） |
| 东北林沼、中亚沙漠、河西戈壁 | **不硬扣** → 第二层 ×0.6 |

## 二、边界从哪里来（必须用界城环线）

### 2.1 界城环线 ≠ `getRegion()` 多边形

项目里有两套边界：

| 名称 | 代码位置 | 用途 |
|------|----------|------|
| **界城环线** | `RegionSystem.ts` → `REGION_BOUNDARY_LOOPS` | zoom=6 文化区虚线；**本测算的面积范围** |
| **判区多边形** | `RegionSystem.ts` → `REGIONS` + `getRegion()` | 据点 `region` 缺省时的坐标判区 |

二者形状 **接近但不等同**。  
**可玩陆地份额** 以 **界城环线围合多边形** 为准（锚点 `cityId` 取 `cities_v2.ts` 当前坐标）。

### 2.2 环线锚点一览

锚点定义见 `src/systems/RegionSystem.ts` 文件头注释与 `REGION_BOUNDARY_LOOPS` 数组。  
修改界城锚点后 **必须重跑** `npx tsx scratch/region_playable_area.mjs`。

### 2.3 环线重叠（重要）

14 条环线 **共用界城**（襄平、宗谷、星主厅、哈密卫等），围合区域 **互相重叠**。  
因此：

- 各区的「可玩份额」是 **独立环线内的蒙特卡洛权重**；
- **不能**把 14 区份额简单相加当作互斥国土面积；
- **不能**用「某点只属于一个环线」做全球互斥分区（除非另立 Voronoi 规则，当前未采用）。

现有据点统计仍用 **`getCityRegion()`**（与游戏运行时一致），与环线面积口径 **分离**。

## 四、第二层：局部历史密度折中

在 **几何建议** `geometricSuggested = round(T × playableShare)` 之后，**按区局部**乘系数（**非**全局重归一化）：

```
auditSuggested[R] =
  geometricSuggested[R] × 0.6     若 R ∈ {东北, 中亚, 河西}
  geometricSuggested[R] × hanBoost  若 R ∈ {中原, 南方, 北方}（见脚本 HAN_BOOST）
  geometricSuggested[R]           其余
```

| 文化区 | 系数 | 典型验算（几何 → 审计） |
|--------|------|-------------------------|
| 东北 | 0.6 | 66 → **40**（≈ 现有 40） |
| 中亚 | 0.6 | 65 → **39**（≈ 现有 36） |
| 河西 | 0.6 | 56 → **34**（≈ 现有 33） |
| 中原 | 1.2 | 68 → 82（汉地仍偏紧，现有 79 为健康锚） |
| 南方 | 1.15 | 61 → 70 |
| 北方 | 1.1 | 60 → 66 |

**652 城现版** 是项目主人排布的健康锚；审计建议用于看 **趋势**（如草原 77 vs 几何 40），不是机械删城令。

## 五、算法步骤（AI 照做即可复现）

```
对每个文化区 R（共 14 个）：
  1. 从 REGION_BOUNDARY_LOOPS 取锚点 cityId → cities_v2 坐标 → 闭合成多边形 ring[R]
  2. 取该区蒙特卡洛包围盒 bbox[R]（见脚本常量）
  3. 重复 N 次（默认 N = 800_000）：
       在 bbox[R] 内均匀随机 (lat, lng)
       若 pointInRing(lat, lng, ring[R])：
         若 isHardUnplayable(R, lat, lng) → unplay++
         否则 → play++
  4. playableShare[R] = play[R] / Σ play
  5. geometricSuggested[R] = round(T × playableShare[R])
  6. auditSuggested[R] = 局部系数折中（§四，非全局重分配）
```

**运行**：

```bash
npx tsx scratch/region_playable_area.mjs
```

| 文化区 | 系数 | 现有 | 审计建议 | 差 | 解读 |
|--------|------|------|----------|-----|------|

**真实性与游戏性综合判断**：

- **东北 / 中亚 / 河西**：不要用掩膜死磕；**652 现版在这三区已接近最优**。
- **审计偏多 ≠ 删城令**：草原 77、青藏 61 等仅作**审视线索**；**默认保留**，除非硬伤（见 §八）。
- **汉地**：几何偏低，**优先向北方、南方增补**，不必为配平删中原。

## 八、数据维护策略：增补优先，删留仅硬伤（2026-06-17 立）

> **项目主人定调**：现版 **652 城** 是多年心血的**健康锚**；大量删点会导致旗号、五文件、精锐、道路心血白费。**默认少删、可向偏少区补点**，全图总量可缓慢上浮（如 680–720），仍守 ≥50 km 与史料。

### 8.1 何时 **补**（优先）

| 信号 | 做法 |
|------|------|
| 审计 `auditSuggested` **明显高于** `currentCities` | 在 **北方、南方、中原** 等汉地缓补；史料 + ≥50 km + FactionEditor 五文件 |
| 某区势力/番号缺锚点 | 单点论证后补，不批量 |
| 道路 | **主人手画**；补点不触发 AI 改 `VectorRoadData.ts` |

### 8.2 何时 **删**（极少、单次、可说明）

| 硬伤 | 说明 |
|------|------|
| 两据点 **< 50 km** 且无删留裁定保留双方 | 删小留大（§2.1） |
| **无史料**、1900 后地名、纯凑数 | 宁删不胡编 |
| 主人 **点名** 撤销 | 执行并同步五文件 |

**禁止**：因 `geometricSuggested` / `auditSuggested` **配额偏多** 而批量删城（尤其草原、青藏、**日本**）。

### 8.3 日本专区（2026-06-17）

**现况**：`getCityRegion` 统计 **31 城**（含界城钓鱼岛、宗谷，及北海道白老/胜山馆/根城，精锐据点如小田原/名张等）。

**近期已做大规模精简**（勿重复删）：

- 已回滚 **2026-06-16 令制国 circuit 批量**（约 39 个 `_d` 扩点，仅留 `hojo_d`、`iga_d` 等精锐相关）。
- 已删 **明治后北海道** 据点，仅恢复 **宗谷** 界城。
- 壹岐改回 `yizhi`@原之辻，未恢复令制国铺点。

**审计对照**：几何建议约 **20** 城 → 现版 31 **仍略高于纯面积模型**，但：

- 日本环线扣海后陆地有限，**20 过稀**，难以承载战国—幕藩多势力与远征节点；
- 31 含 **文化中心京都**、**界城/环锚**、**南北岛链**，属 **刚删完后的合理密度**；
- **结论：日本不多，不宜再删**；若动刀仅限 \<50 km 硬伤或主人点名，**不**为凑 20 而砍。

## 十、AI 操作红线

1. **禁止** 为东北 / 河西 / 中亚 **新增**几何扣除掩膜（见 §3.7）。
2. **禁止** 把 `auditSuggested` 当作必须删到的精确整数；**禁止**因配额偏多批量删城（§八）。
3. **禁止** 在未获主人点名时 **删减日本** 据点（§8.3：刚完成令制国回滚与北海道清理）。
4. **禁止** 用 `REGIONS` 多边形代替界城环线。
5. **禁止** 为凑数挪动 lat/lng 或胡编据点。
6. **禁止** 修改 `VectorRoadData.ts` / 自动连线。
7. **禁止** `git checkout/restore/reset`。
8. 改第一层掩膜或第二层系数 → 改脚本 → 重跑 → 更新本文 §七 表格。
