# 立绘调校「调好又恢复」问题 — 根因与解决方案

> 日期：2026-06-27
> 涉及文件：`src/config/PortraitAdjust.ts`、`src/ui/CombatUI.ts`、`src/config/portrait_canonical.ts`、`src/data/portrait_adjust.ts`

## 一、现象

用 F2 调好某将领（如耶律倍）的立绘缩放/位置，按 Enter 保存后，**下一场战斗又恢复到原始位置**；
有时调好的值会变成另一个将领的值。几百个将领反复出错。

## 二、根本原因：canonical「内容去重」把不同将领塞进同一个存储格子

### 调校的存储结构
`src/data/portrait_adjust.ts` 的 `DEFAULT_PORTRAIT_ADJUST`：
- `folders`：文件夹级默认（如 `/assets/NORTHEAST/`）
- `images`：单张覆盖，键为立绘路径，值为 `{ scale, offsetX, offsetY }`

### canonical 去重表
`src/config/portrait_canonical.ts`（脚本 `npm run portrait:build-canonical` 按**文件内容哈希**生成）。
内容相同的多张立绘被归并到同一个「代表路径」。例如东北区很多将领开局都在用同一张占位图，于是：

```
/assets/NORTHEAST/dongdan_yelvbei.png  →  /assets/NORTHEAST/NORTHEAST_field.png
/assets/NORTHEAST/其他将领.png          →  /assets/NORTHEAST/NORTHEAST_field.png
... （几十个将领映射到同一代表路径）
```

### 旧逻辑（出 bug 的版本）
- **存盘键** `correctorSaveKey()` = `toCanonicalPortraitPath(显示路径)` → 存到**代表路径**
- **读取键** `resolvePortraitAdjust()` = `toCanonicalPortraitPath(显示路径)` → 读**代表路径**

单看一个将领是自洽的，但因为多个将领共用同一个代表路径：

```
调耶律倍 → 写入 images["NORTHEAST_field.png"] = A
再调另一个东北将领 → 覆盖 images["NORTHEAST_field.png"] = B
下场耶律倍读 images["NORTHEAST_field.png"] = B   ← 变成别人的值（“恢复/变了”）
```

**几百个共享代表路径的将领，全在抢同一个格子，互相覆盖。**

### 雪上加霜：去重表会过期
`portrait_canonical.ts` 是某一时刻（2026-06-25）生成的静态文件。之后用户给将领**绑定了独占新图**，
文件内容已唯一，但去重表仍把它当作旧的共享图 → 即使按代表路径存读，仍指向错误的共享格子。

## 三、解决方案：按「立绘自身路径」存取，canonical 仅作读取兜底

核心思路：每个立绘按**自己的真实路径**独立存一格；canonical 只在自身路径无记录时作为兜底，
保证历史上已按代表路径存的几百条调校不丢失。

### 读取优先级（`resolvePortraitAdjust`）
```
① images[自身路径]              （每将领独立，最高优先）
② images[canonical 代表路径]    （旧数据兜底，历史调校仍生效）
③ folders[文件夹]               （文件夹级默认）
④ 中性值 { scale:1, offsetX:0, offsetY:0 }
```

### 存盘键（`correctorSaveKey` / `correctorSaveKeyForSide`）
直接用立绘自身路径（`correctorPath()`），**不再**归并到 canonical。
待绑定新图用 `destPath`（最终落盘路径），同样是自身路径。

### 配套改动
- 删除 `applyCorrectorPreview` / `syncCurrentCorrectorDraftToData` 里原先「删除命名条目以归并到 canonical」
  的死代码（与新模型相反）。
- `bindSelectedPortrait` 读取源图调校改用 `resolvePortraitAdjust(sourcePath)`（自身优先）。

## 四、修复后的行为

| 场景 | 结果 |
|------|------|
| 几个将领**共用同一个文件路径** | 天然共享同一格（同 key）。调一次，全员生效。**符合预期，无 bug** |
| 几个将领是**不同文件**（内容碰巧相同） | 各按自身路径独立存取，**互不覆盖** |
| 绑定独占新图后调校 | 存到该将领自身路径，**永久生效，不再被别人覆盖** |
| 历史上已按代表路径存的调校 | 经 ② 兜底继续生效，**不丢数据**；一旦该将领被单独调过，自身路径条目接管 |

## 五、重要结论：不需要每人独占一张立绘

- **可以**继续让多个将领共用同一张图（同一文件路径），调一次全员生效。
- 只有当你希望共用同一文件的将领拥有**不同**位置时，才需要拆成不同文件
  （一个文件只有一份调校）。

## 六、对现有数据的影响

**无损**。本方案不改写、不删除 `portrait_adjust.ts` 里的任何历史条目：
- 已按命名路径存的条目 → 读取①直接命中。
- 已按代表路径存的条目 → 读取②兜底命中。
- 新调校 → 写入自身路径，自然遮蔽旧的共享格子，从此独立。

## 七、可选的后续清理（非必须）

若想彻底摆脱过期去重表，可重新生成 canonical：`npm run portrait:build-canonical`。
但**不必须**：本方案已让自身路径优先，过期映射不再影响新调校。重新生成前应评估
是否会让某些「仅按代表路径存」的历史调校失去兜底（重生成后代表路径可能变化）。
