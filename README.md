1. 游戏概述

游戏类型： 大战略历史模拟游戏 (Grand Strategy / 4X SLG)

运行平台： Web 浏览器 (PC 端为主，兼容移动端缩放交互)

视觉核心： 基于真实地理（Google Maps / 可缩放宏观地图）的无缝大世界体验。

设计理念： 系统机制先行，历史剧本涌现。 以“点-线-面”的地理博弈为核心，拒绝生硬切图，让玩家在每一次缩放中体验从“运筹帷幄”到“金戈铁马”的战略快感。

2. 核心技术架构选型

底层引擎： 基于地理信息系统 (GIS) 瓦片地图引擎（Google Maps API / Mapbox GL JS）。

渲染层： HTML5 Canvas / PixiJS (WebGL)。地图引擎仅负责地形背景底图和坐标系转换；所有的据点、势力多边形（Voronoi）、部队移动、战斗动画全部在叠加的 Canvas 层上用代码动态绘制，确保千人同屏不卡顿。

数据结构演进： * 城市据点：核心节点 (Nodes)。

领土划分：基于据点实时计算的泰森多边形 (Voronoi Diagram)。

寻路数据（核心）：初期采用“无向图 (Graph)”，中后期平滑过渡为“带权重的栅格矩阵 (Weighted Grid Matrix)”。

3. 核心机制设计 (Rules of Physics)

3.1 军队行军与寻路机制（技术演进版）

为了平衡开发周期与游戏最终的策略深度，部队的行军机制将分为两个阶段实现：

前期阶段（点线模式 / Node-Graph）：

机制： 城市与城市之间存在预设的连线（平原路、山路、水路）。部队只能沿着这些连线移动。

设计意图： 极大地降低开发初期的寻路算法复杂度。战略重点在于“卡脖子”（扼守关键节点或渡口），类似于《三国群英传》的大地图调兵。

后期阶段（地形权重矩阵 / Weighted Grid）：

机制： 将整个地图栅格化。每个格子拥有不同的地形阻力（道路=1，平原=2，森林=4，高山=不可通行）。部队可以脱离既定路线，在势力范围内或野外自由寻路（基于 A* 算法）。

设计意图： 解锁真正的硬核战术。玩家可以实现“绕后包抄”、“野外森林伏击”、“大迂回切断补给”等高阶战役微操，向《全面战争》的战略深度靠拢。

3.2 “面”的控制：势力范围 (ZOC)

机制： 玩家占领一个“点”，系统立即利用 Voronoi 算法，将该点周围的区域划分为玩家的势力颜色（半透明色块覆盖地图）。

战略纵深与摩擦力： 敌军进入我方色块范围时，视野暴露，且移动速度降低 30%，粮草消耗翻倍。如果部队离己方据点过远，且后方补给路线被切断，部队将陷入“断粮”状态，每秒损失士气和兵力。

3.3 无缝缩放战斗系统 (Seamless Combat)

坚持“绝不切图 (No Scene-Switching)”的原则，采用缩放联动 (Zoom-Linked LOD) 来呈现战斗。

接战判定： 两个不同阵营的部队坐标在地图上重叠或碰撞（距离小于阈值），即进入交战状态。

表现形式（缩放联动）：

宏观视野 (Zoom Out - 统帅视角)： 地图缩小至国家版图级别。玩家看到的是红蓝两个色块或光点在交汇，伴随轻微的屏幕震动和细小的跳动伤害数字（如 -150）。

中观视野 (Zoom Mid - 将军视角)： 放大至省州级别。光点展开为兵种图标，显示双方的士气条和兵力条。

微观视野 (Zoom In - 战场视角)： 放大至局部地形（如某条河畔）。显示部队的阵型，并在交战点上方弹出武将单挑、释放技能（如火计爆发）的绚丽 Canvas 特效，伴随金戈铁马的音效。

结算系统： 采用异步回合制心跳计算。
单次伤害 = (兵力 × 基础攻击力 × 武将统率加成 × 地形克制系数) - 敌方防御计算

4. 游戏核心循环 (Core Loop)

【种田与分配】 在安全的势力范围内，升级据点，收集粮草与金币。

【情报与博弈】 观察地图战争迷雾边缘，分析敌军动态。

【出征与风险】 派遣武将，携带粮草向目标进军。路途耗时即是风险期。

【碰撞与扩张】 在无缝大地图上与敌军发生实时碰撞结算。

【版图重构】 胜利后占领据点，系统重新计算 Voronoi 多边形，领土边界向前推进，回到第 1 步。

5. “机制先行”的历史事件引擎 (Event Engine)

游戏本体是一个纯粹的“地理与数值沙盒”。所有的历史事件通过外部脚本“有条件地”注入。

触发机制： 监听器根据“时间 + 地理 + 人物状态”触发脚本（如：占领洛阳触发迁都事件）。

涌现式叙事： 依靠拟真的地理阻力（如长江天险、蜀道难）和补给线机制，让玩家的战略选择自然而然地重现历史上的“南北对峙”或“守险拒敌”，而非依靠纯文本强行推动。

6. 敏捷开发路线图 (Roadmap)

第一阶段：MVP 沙盒（点线架构 + 无缝碰撞验证）

第二阶段：Alpha 版本（缩放联动 + 系统填充）

开发重点： 完善缩放联动 (LOD) 视觉表现；接入外部 JSON 配置的真实城市与武将数据；实装粮草消耗（摩擦力）与迷雾系统。

开发重点： 废弃初期的硬编码连线。引入地形权重矩阵 (Grid Matrix) 与 A* 寻路。部队可以脱离道路进行自由的野外机动。实装地形对行军和战斗的具体数值影响（如：骑兵在森林中减速且降低攻击力）。

第四阶段：正式上线准备

开发重点： 引入事件引擎，编写历史剧本脚本。深度优化 Canvas 渲染性能（处理矩阵化后海量的寻路计算）。

# MAPWAR - 历史地理仿真沙盒 (Historical Geospatial Sandbox Simulator)

> **核心定位**: 2D 世界地形模拟器 | 伪直播架构 (Pseudo-Live) | 指令驱动 (Instruction Driven)

MAPWAR 不仅仅是一个游戏，而是一个**"世界地形模拟器"**。它致力于通过极简的指令流，在客户端还原宏大的历史地理变迁。

## 1. 核心哲学 (Core Philosophy)

###  沙盒模拟 (Sandbox Simulation)
*   **非游戏 (Not a Game)**: 我们不追求RTS的微操快感，而是追求《文明》式的宏观历史推演。
*   **视觉风格**: 2D 策略风格 (Civ-like)，专注于地形、国界与战略态势的可视化。

###  伪直播架构 (Pseudo-Live Architecture)
*   **挑战**: 如何用最低成本实现全球观众同步？
*   **解决方案**: **指令驱动 (Log-Driven)**。
    *   **服务器**: 不传输实时画面/位置帧，只广播 **"指令序列"** (如 `Year:202, Faction:Han, Action:Sortie(Target:Xuzhou)`)。
    *   **客户端**: 仅仅是一个**"播放器"**。接收指令后，基于确定性的本地逻辑（A*寻路、战斗公式）重现画面。
    *   **优势**: 极大降低带宽（文本流 vs 视频流），单台低配 VPS 即可支撑全球同步。

### 🤖 都城中心制 AI (Capital-Centric AI)
为了配合"指令驱动"架构，我们采用了极简的 AI 模型：
*   **单一都城 (Single Capital)**: 每个势力拥有一个逻辑核心。
*   **出征模式 (Sortie Mode)**: 军队平时**驻扎（隐藏）**在都城。只有当收到攻击指令时，才作为一次"事件"被生成，执行完任务即解散或回城。
*   **价值**: 这种设计消除了海量的"漫游状态"同步需求，让每一条指令都清晰明确。

## 2. 技术架构 (Technical Architecture)

### 🗺️ 全球地形 (Global Terrain)
*   **无缝滚动**: 基于 `GameMap.ts` 的无边界设计，地形本身即为全球无缝。
*   **GIS 渲染**: 实时读取 Terrarium 高度数据，通过 `HillshadeLayer` 进行高性能 GPU 渲染（内联数学优化）。

### ⚔️ 战斗演化 (Combat Evolution)
*   **微观表现**: 虽然宏观上是指令，但微观上仍保留了细腻的兵棋战斗（冲锋、插空站位、侧翼包抄）。
*   **确定性**: 所有的战斗结果由数值和逻辑决定，确保所有客户端看到的"演出"是一致的。

## 3. 系统模块 (System Modules)

### 核心系统 (Core Systems)

#### 1. 军团与单位系统 (Legions & Units)
项目有四大战斗单位：玩家、部队、盗贼、城市。
*   **城市 (City)**: 静态据点，拥有固定坐标、势力归属和驻军。城市是兵力的来源。
*   **军团 (Legion)**: 移动战斗单位。在"都城中心制"下，军团平时驻扎在都城，仅在执行指令时生成。
*   **征兵机制**: 军团创建时从同阵营城市抽调兵力，城市保留基本守备军。

#### 2. 宏观战略战斗 (Grand Strategy Combat)
*   基于 Zoom 9 级别的宏观战略。
*   无微操，通过 UI 面板和自动化逻辑推进战斗（先登 -> 陷阵 -> 斩将 -> 夺旗）。

#### 3. 视图与网格 (Map & Grid)
*   1个六边形(Hex) ≈ 15-20公里。
*   城市是一个逻辑点，无"关内/关外"物理区分。
*   **作战单位**: 
  - 最小作战单位为**军团 (Legion)**（万人级），而非单兵。

#### 4. 势力 (Factions)
包含几个历史势力（华夏、中华、天朝，满洲等）。

#### 5. 历史事件分类 (Historical Event Types)
为支撑两千年的宏大历史推演，引擎将所有复杂的历史事件高度抽象为 **三种核心逻辑**，即可覆盖 99% 的史料记载：
*   **攻城战 (`siege`)**: 涉及城市所有权的变更。不仅可以用于常规的攻城略地，也可以完美模拟**起义与割据**：起义军（如陈胜吴广、黄巾军）直接在目标城市作为攻击方生成，若攻城成功则该城市势力易手为起义军势力。
*   **野战 (`field_battle`)**: 以消灭敌方有生力量为目的的遭遇战或合围战。两只独立军团在特定经纬度 (`location`) 碰撞，失败方被全歼，但不改变脚下城市的归属。同样适用于起义军与官军的野外决战。
*   **叙事 (`narrative`)**: 战略决变、天灾人祸、刺杀等不开战的图文事件。

## 6. ✅ 近期系统优化 (Recent Optimizations)

为提升系统稳定性与逻辑一致性，已完成以下核心优化：

1.  **渲架性能大幅提升 (Rendering Performance)**:
    -   **HillshadeLayer 重构**: 对核心地形渲染循环进行了深度优化。
        -   **内联数学运算**: 将每像素必须的 9 次高度获取函数调用全部展开为内联数学公式，消除了数百万次函数调用开销。
        -   **路径分离**: 将“彩色地形”与“灰度地形”拆分为两个独立的渲染循环，移除了循环内部的条件分支判断。
    -   **效果**: 显著减少了拖动地图时的掉帧现象，提升了渲染流畅度。

2.  **修复历史事件冲突 Bug (Fixed Event Conflicts)**:
    -   **问题**: 旧系统使用 `年份_季节_类型` 作为唯一键，导致同一季节发生多个同类事件时（如多场攻城战），后续事件会被系统忽略。
    -   **解决**: 实现了确定性事件 ID 生成 (`Stable ID Generation`)。系统现在根据事件文件中的索引生成唯一 ID，确保所有并发事件都能被正确触发和追踪。

3.  **统一战斗与参战逻辑 (Unified Combat Logic)**:
    -   **问题**: 攻城战 (`Siege`) 与野战 (`Field Battle`) 自动寻路距离判断逻辑分散且不一致。
    -   **解决**: 创建了 `CombatUtils.checkPlayerParticipation` 统一接口。
    -   **效果**: 无论战斗类型如何，系统现在使用统一的距离

4.  **配置集中化 (Centralized Configuration)**:
    -   消除多处“魔法数字”（如 `9999` 自动加入距离），统一移至 `GameConfig.ts` 进行管理，提升代码可维护性。

5.  **数据持久化 (Data Persistence)**:
    -   **功能**: `HistoricalEventManager` 现在会自动将已触发的历史事件状态保存到本地存储 (`localStorage`)。
    -   **效果**: 即使页面刷新或重新加载，系统也会记住哪些历史事件已经发生过，防止重复触发。
